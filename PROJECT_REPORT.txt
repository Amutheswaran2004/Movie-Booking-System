PROJECT REPORT: Movie Booking System

Project Name: Movie Booking System
Project Type: Backend REST API (Java / Spring Boot)

---
## 1. Project Overview
- What the project is: A robust backend system designed to handle the core operations of a movie booking application, providing REST APIs to manage movies, users, and ticket bookings securely.
- Problem it solves: It digitizes the manual movie booking process, handling tasks such as checking seat availability, booking tickets, updating theater capacities, and maintaining user records centrally while protecting the endpoints against unauthorized access.
- Real-world use case: Can be used by cinemas or online ticketing platforms (like BookMyShow or Fandango) as the underlying engine to power booking procedures, coordinate ticket inventory in real-time, and store customer data safely.
- Why this project is useful: It ensures race-condition management (total tickets remaining updating accurately), provides clean REST endpoints for frontend consumption, and uses production-ready technologies like PostgreSQL, Spring Boot, and robust JSON Web Token (JWT) Security.

- Backend technologies: Java 17, Spring Boot 4.0.1, Spring Web (REST API layer), Spring Data JPA (Hibernate/ORM layer)
- Security: Spring Security, JSON Web Tokens (JJWT library), BCrypt Password Encryption
- Database: PostgreSQL
- Tools and IDEs used: Maven (Build Tool), Postman (for API Testing - standard practice), Git/GitHub (Version Control).

## 3. Project Architecture
- Overall structure: The project follows a standard N-Layered Architecture common in enterprise Spring Boot applications:
  1. Controller Layer (`controller`): The entry point for HTTP requests. Maps URLs to specific operations.
  2. Security Layer (`security`): Filters incoming requests, validates JWT tokens, extracts User principals, and locks down protected endpoints.
  3. Service Layer (`service`): Contains the core business logic. Acts as a bridge between Controllers and the Database layer.
  4. Repository Layer (`repository`): Data Access layer extending Spring Data JPA interfaces to communicate directly with PostgreSQL.
  5. Model Layer (`model`): Contains JPA Entities that map directly to PostgreSQL database tables.
  6. DTO Layer (`dto`): Data Transfer Objects used to receive structured JSON payloads from users without exposing internal database entities.
- Flow of data (step-by-step):
  1. Client hits `/register` sequentially hashing a secure password into the DB.
  2. Client sends a POST request with credentials to `/login`.
  3. The API validates the identity, generating and returning a stateless JWT token.
  4. The client attaches this 'Bearer' token in the HTTP Headers for subsequent requests (e.g., POST `/api/v1.0/moviebooking/Avatar/book`).
  5. The `AuthTokenFilter` intercepts the request, validates the token via `JwtUtils`, and grants network access.
  6. The `MovieController` maps the DTO payload to the `MovieService`.
  7. The Database saves the ticket and updates available capacity sequentially.

### File: JwtUtils.java
- Purpose: Component responsible purely for Token manipulation.
- Role: Executes methods natively mapping cryptographic `Keys.hmacShaKeyFor()`.

### File: AuthTokenFilter.java
- Purpose: Middleware network filter boundary implementations.
- Role: The `AuthTokenFilter` executes "Once Per Request," extracting the "Bearer" mapped header, validating it, and forcibly establishing a basic `UsernamePasswordAuthenticationToken` context.

### File: MovieController.java
- Purpose: Handles all incoming HTTP requests.
- Role: Acts as the primary REST API delegator mapping endpoints.
- Why needed: Contains the isolated `/login` execution pipeline doing manual password validation and returning JWT logic natively, injects the explicitly created `PasswordEncoder` safely encoding registrations.

### File: MovieService.java
- Purpose: Contains all primary business logic and capacity math.
- Role: Orchestrates real-time seat tracking via Java streaming. Fully recalculates ticket availabilities toggling states based on factual mapped history arrays mapping.

### File: Movie.java, Ticket.java, User.java

### File: MovieRequest.java & TicketRequest.java
- Purpose: Distinct Payload Data Transfer Objects generating inputs masking raw DB PK constraints entirely.

### File: LoginRequest.java & JwtResponse.java
- Purpose: DTO mechanisms designed definitively for Security interactions bridging raw login queries to definitively packed JWT token responses containing contextual user IDs natively protecting backend structures.

### File: MovieRepository.java, TicketRepository.java, UserRepository.java

## 6. API Design
- **POST `/api/v1.0/moviebooking/register`** (Public)
  Purpose: Creates entirely new user authorization profiles. (Hashes password).
- **POST `/api/v1.0/moviebooking/login`** (Public)
  Purpose: Verifies credentials manually parsing user from DB, checks password, and distributes a secure JWT.
  Request format: JSON Object (`LoginRequest` schema).
  Response format: JSON Object (`JwtResponse` mapping exact token hash strings).
- **GET `/api/v1.0/moviebooking/all`** (Protected)
  Purpose: Retrieves all available movies physically mapping across JSON. Requires `Authorization: Bearer <token>`.
- **POST `/api/v1.0/moviebooking/{moviename}/book`** (Protected)
  Purpose: Reserves tickets mapping logic dynamic capacities strictly securely across active contexts natively utilizing token verifications natively.

*(All other endpoints `/addMovie`, `/search/`, `/booked-seats-text/` require valid injected 'Bearer' tokens)*

---
## 7. Key Features
- **Stateless JWT Authentication:** Implemented a highly scalable security layer utilizing Spring Security and `jjwt`. Clients are issued a mathematically signed token upon `/login` requiring an `Authorization: Bearer` header on subsequent internal API requests, creating a robust, sessionless server architecture.
- **BCrypt Password Cryptography:** Defensively secures DB parameters. When `/register` hits the application, `PasswordEncoder` securely hashes user payloads preventing any plain text exposures natively bridging PostgreSQL structures safely.
- **Dynamic Capacity Management:** Whenever a ticket is booked securely via a signed token, the backend directly recalculates absolute seating capacities across complex Stream structures toggling schemas mapping cleanly toward "SOLD OUT" flags inherently circumventing negative overlaps.

## 8. Challenges & Solutions
- Challenge: Ensuring APIs remain strictly secure without breaking standard stateless REST architectural laws heavily relying on heavy Session limits.
  Solution: Bridged the complex `OncePerRequestFilter` native class dynamically constructing an `AuthTokenFilter`. Overrode explicit explicit Spring limits avoiding Session Creation mappings enforcing a total stateless JSON-driven network architecture mapping user context explicitly via tokens solely on isolated logic passes.
- Challenge: Dealing with critical overlapping logic mapping across ticket database overwriting limiting accurate thresholds manually.
  Solution: Reconfigured system architectures applying programmatic calculations dynamically aggregating entire total-historical instances inside `MovieService.updateTicketStatus` outputting factual totals avoiding database-level concurrency flaws.
